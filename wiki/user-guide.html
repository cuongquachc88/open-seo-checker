1. Install

The repo ships a pnpm workspace with two packages (@oseo/api and @oseo/web) plus a thin orchestrator script for development. You need:

  • Node.js ≥ 18
  • pnpm (install with npm i -g pnpm)

First-time bootstrap:

$ git clone <repo> && cd open-seo-checker
$ pnpm install                # deps + Playwright Chromium postinstall
$ pnpm build                  # compile api + web
$ pnpm server                 # production: API + SPA on :7437
The first run installs the Chromium-headless that powers JavaScript rendering (npx playwright install chromium). It can take a minute or two; subsequent runs are instant.

2. Launch

Pick your flavour:

Production pnpm server

Single port :7437, serves the API and the dashboard SPA, auto-opens Chrome. Best for one-time audits.

Dev orchestrator pnpm start:sh

Shared wordmark + colour-tagged backend & frontend logs, readiness probes, Ctrl+C graceful shutdown. Best while iterating. The backend and frontend processes each print their own solid-colour role banner (blue BE, magenta FE).

One-click launcher

./open-seo-checker.sh on macOS / Linux, open-seo-checker.bat on Windows. Auto-builds if needed and opens Chrome.

Second-screen monitor

pnpm monitor runs alongside pnpm start:sh in a separate terminal to keep an eye on the log streams without scrolling.

When pnpm start:sh is up, each role prints its own banner in the shared terminal. The backend banner is solid ● blue and the frontend banner is solid ● magenta, so you always know which process produced a log line.

3. Run a crawl

The New Crawl page acts as a wizard. The minimum: a starting URL, a spider or list mode, and a URL / depth cap. Most of the other knobs map directly to commands you would expect from a mature crawler.

  • Mode — Spider follows links from a seed URL; List only crawls the URLs you typed.
  • Max URLs / Max depth — hard ceiling to keep a runaway crawl in check.
  • Threads — number of concurrent fetch workers (1-32).
  • Render JS — ask Playwright to render with a headless Chromium before extracting links / titles / structured data.
  • Robots.txt — enable by default; turn off only if you know what you are doing.
  • Subdomains — treat blog.example.com as part of example.com or as external.
  • Follow redirects — the crawler records the full chain (max 10 hops with cycle detection).

4. Read a report

Once a crawl finishes, you land on the Overview tab. The hero health gauge is a 0-100 composite with per-category breakdowns; quick-wins tiles surface the highest-impact issues.

The sidebar links every persistent artefact to the same already-known view — clicking Crawl Runs opens the runs table, clicking Reports opens the export centre.

5. Issues tab

The Issues tab groups the open problems by category (titles, meta, headings, images, links ...) and severity (high / medium / low). Every issue includes a how to fix hint and a deep link to the URL it was found on. The "by URL" matrix at the bottom shows which pages contribute to each issue.

Filter the table to combine category × severity. Use ⌘ + F to search URLs and titles from any tab.

6. Sitemap studio

Two roles for sitemaps here: crawl-derived from the data already in your stored run, or live discovery by pointing the crawler at a website and asking it to print XML. 

$ oseo sitemap https://example.com --output ./sitemap.xml --max-urls 5000

The dashboard tab renders the resulting XML with copy and download buttons. Image entries are emitted under the image: namespace whenever the crawler found <img> tags with valid src.

7. Comparing runs

Pick a baseline run, pick a current run, and the dashboard diffs them: added URLs, removed URLs, changed titles, changed indexability. Useful for "what slipped since last month" reviews.

Make sure baselines live on the same database. The Compare tab reads crawls/<name>.db files; if you've moved files around, re-import from the runs table.

8. AI Insights

  1. Open Settings.
  2. Pick a provider under Default AI provider (OpenAI, Anthropic, Gemini, Kimi, MiniMax, or Ollama for self-hosted).
  3. Enter your API key on the same card.
  4. Save. The Insights tab automatically hides its empty state.

The chat pre-fills four prompt templates you can fire against the stored crawl: "Page-level content audit", "Top 20 quick wins", "Schema opportunities", "Internal-link gaps". Edit them in place or write your own.

9. CLI mode

If you don't need the dashboard, every command above has a CLI equivalent. The oseo binary is exposed when you install @oseo/api. 

$ oseo serve                       # same as `pnpm server`
$ oseo crawl https://example.com --max-urls 5000 --render
$ oseo sitemap https://example.com  --output sitemap.xml
$ oseo compare crawls/old.db crawls/new.db
$ oseo logs  /var/log/nginx/access.log --bot googlebot
$ oseo health crawls/latest.db

Each command runs the same analyzer stack the dashboard uses; the output is the same SQLite database the dashboard reads. You can mix CLI and dashboard freely.

10. Troubleshooting

The dashboard loads but every panel is empty

You may have just deleted the crawls/ folder. Cached data lives there; restart a crawl to repopulate. Or restore a backup.

JS rendering hangs forever

Playwright's Chromium may be blocked by corporate firewalls on first download. Run npx playwright install chromium manually if the postinstall couldn't reach the binary CDN.

An AI provider returns 401

Keys are stored in your browser's local storage, not on disk. Signing out / clearing site data wipes them. Re-enter and save again from Settings.

Two or more crawls are blocking the same port

Kill a stale process with lsof -ti:7437 | xargs kill -9. The dev orchestrator warns at startup if either port is already bound.
