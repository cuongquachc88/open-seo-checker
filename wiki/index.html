Open SEO Checker is a free, open-source, cross-platform website crawler and SEO auditing tool with a local CLI and a built-in dashboard. Crawl audits, content scoring, AI insights, schedule recurring crawls — all from your own machine.

Cross-platform

Quick start #

One-command installer. Runs Node + pnpm checks, builds the project, and drops a desktop shortcut that starts the server and opens the dashboard with a single double-click. 
$ ./install.sh        # macOS / Linux
$ install.bat         # Windows   (double-click works too)
Re-run any time to refresh the shortcut or rebuild.
  • macOS shortcut: ~/Desktop/Open SEO Checker.app
  • Linux shortcut: ~/Desktop/Open SEO Checker.desktop
  • Windows shortcut: %USERPROFILE%\Desktop\Open SEO Checker.lnk

Deep-dive (prerequisites, troubleshooting, uninstall, verifying the install from a fresh curl) lives in docs/INSTALL.md at the repo root and is bundled with every GitHub release.

You need Node.js ≥ 18 and pnpm (install with npm i -g pnpm).

$ git clone <this-repo> && cd open-seo-checker
$ pnpm install                # deps + Playwright Chromium postinstall
$ pnpm build                  # compile api + web
$ pnpm server                 # API + dashboard on :7437, auto-opens Chrome

# Or live dev (Vite HMR, colour-coded BE+FE logs):
$ pnpm start:sh
$ pnpm dev:web                # only the dashboard on :5173

# One-click launcher:
$ ./open-seo-checker.sh        # macOS / Linux
$ open-seo-checker.bat         # Windows

# Quality:
$ pnpm test:unit               # 26 vitest specs
$ pnpm test:e2e                # 23 Playwright specs
$ PW_BOOT=1 pnpm test:e2e      # spawn dev:sh itself, then tear down
The first run installs the Chromium headless that powers JavaScript rendering (npx playwright install chromium). It can take a minute or two; subsequent runs are instant.

Every command has both a oseo binary and a pnpm shortcut.

Features #

Core audits

  • Crawl engine: spider + list mode, robots.txt compliance, custom UA, depth, concurrency, redirect chain handling, sitemaps.
  • On-page: titles, meta, headings, canonicals, hreflang, pagination, images, URL structure.
  • Technical: broken links, redirect chains/loops, internal linking, anchor text, security headers, mixed content.
  • Rendering: JS rendering via Playwright, structured-data extraction, AMP structural checks, a11y lints.
  • Content: duplicate / near-duplicate detection, word count, text ratio, custom extraction (XPath / CSS / regex).
  • Integrations: GA4, GSC, PageSpeed Insights, Majestic, Ahrefs, Moz.
  • AI: OpenAI, Anthropic, Gemini, Kimi, MiniMax, Ollama (local).
  • Export: CSV / JSON / XLSX / XML sitemaps + crawl comparison + scheduling.

Workflow extensions

  • Log-file analysis: parse server logs for bot behavior, orphans, crawl budget waste.
  • Backlink analysis: inspect external links pointing to your domain from the crawl data.
  • Keyword extraction: titles / headings / meta.
  • Content scoring: quality, density, readability.
  • Local SEO: NAP, LocalBusiness schema, contact-page signals.
  • E-commerce SEO: Product / BreadcrumbList schema, faceted nav, cart & checkout indexability.
  • AI search visibility: llms.txt generation, E-E-A-T signal checks.
  • SEO health score: 0-100 composite with category breakdowns.
  • Crawl comparison: added / removed / changed URLs between two runs.
  • Scheduling: built-in cron-like scheduler for recurring audits.

User guide users #

Headless-free flow: install → launch → run a crawl → read the report. The dashboard takes care of the rest.

Install

  • Node.js ≥ 18, pnpm
  • First-time pnpm install downloads Playwright Chromium for JS rendering.

Launch

Production · pnpm server

Single port :7437, API + SPA, auto-opens Chrome.

Dev orchestrator · pnpm start:sh

BE+FE colour-coded logs, readiness probes, Ctrl+C graceful shutdown. Each role prints its own banner.

One-click launcher

./open-seo-checker.sh (macOS/Linux) / open-seo-checker.bat (Windows).

Second-screen monitor

pnpm monitor alongside pnpm start:sh.

When pnpm start:sh is up, each role prints its own banner. The backend banner is solid ● blue and the frontend banner is solid ● magenta, so the log line source is always obvious.

Run a crawl #

The New Crawl page is a wizard. Minimum: a starting URL, mode, URL/depth cap.

  • Mode — Spider follows links from a seed URL; List only crawls literal URLs.
  • Max URLs / Max depth — hard ceiling.
  • Threads — concurrent fetch workers (1-32).
  • Render JS — Playwright Chromium before extracting links / titles / structured data.
  • Robots.txt — on by default.
  • Subdomains — internal vs external.
  • Follow redirects — full chain recorded, max 10 hops with cycle detection.

Read a report #

The Overview tab shows a 0-100 composite health gauge with category breakdowns and quick-wins tiles. The sidebar links every persistent artefact to the same old view.

Issues tab #

Groups open problems by category (titles, meta, headings, images, links...) and severity (high/medium/low). Every issue includes a how to fix hint and a deep-link to the offending URL.

Filter the table — category × severity — and search URLs / titles with ⌘ + F.

Sitemap studio #

Two roles: crawl-derived from a stored run, or live discovery by pointing the crawler at a URL and printing XML.

$ oseo sitemap https://example.com --output ./sitemap.xml --max-urls 5000

Image entries are emitted under the image: namespace whenever the crawler found <img> tags with valid src.

Comparing runs #

Pick a baseline run plus a current run and the dashboard diffs them: added URLs, removed URLs, changed titles, changed indexability. Great for "what slipped since last month" reviews.

Make sure baselines live on the same database. The Compare tab reads crawls/<name>.db files; if you've moved files around, re-import from the runs table.

AI Insights #

  1. Open Settings.
  2. Pick a provider under Default AI provider.
  3. Enter your API key on the same card.
  4. Save. The Insights tab automatically hides its empty state.

The chat pre-fills four prompt templates: page-level content audit, top-20 quick wins, schema opportunities, internal-link gaps. Edit in place or write your own.

CLI mode #

If you don't need the dashboard, every command has a CLI equivalent. The oseo binary is exposed when you install @oseo/api.

$ oseo serve                       # same as `pnpm server`
$ oseo crawl https://example.com --max-urls 5000 --render
$ oseo sitemap https://example.com  --output sitemap.xml
$ oseo compare crawls/old.db crawls/new.db
$ oseo logs  /var/log/nginx/access.log --bot googlebot
$ oseo health crawls/latest.db

Each command runs the same analyzer stack the dashboard uses; the output is the same SQLite database the dashboard reads. Mix CLI and dashboard freely.

Troubleshooting #

The dashboard loads but every panel is empty

You may have just deleted the crawls/ folder. Restart a crawl to repopulate, or restore a backup.

JS rendering hangs forever

Playwright's Chromium may be blocked by corporate firewalls on first download. Run npx playwright install chromium manually if the postinstall couldn't reach the binary CDN.

An AI provider returns 401

Keys live in your browser's local storage, not on disk. Clearing site data wipes them. Re-enter and save from Settings.

Two crawls are blocking the same port

Kill a stale process with lsof -ti:7437 | xargs kill -9. The dev orchestrator warns at startup if either port is already bound.

Developer guide devs #

Internals: how the monorepo is wired, where to add new analyzers / routes / AI providers / integrations, how to write both unit (vitest) and end-to-end (Playwright) tests, how to release.

Setup

$ pnpm install
$ pnpm --filter @oseo/api dev     # tsx watch backend at :7437
$ pnpm --filter @oseo/web dev     # vite dev server at :5173
$ pnpm start:sh                  # both at once, BE+FE colour banners

# Quality
$ pnpm test:unit                  # vitest unit suite (api)
$ pnpm test:e2e                   # playwright spec (needs a server)
$ PW_BOOT=1 pnpm test:e2e         # spawn dev:sh, run, teardown

Add a backend analyzer #

  1. Add packages/api/src/analyzer/my-new-check.ts exporting analyzeMyNewCheck(runId: number): void.
  2. Read URLs via getUrls(runId, { isInternal: true }) and produce an array of CrawlIssue.
  3. Call insertIssues(runId, issues) at the end.
  4. Export from packages/api/src/analyzer/index.ts.
  5. Wire from the engine once URL metadata is complete (see packages/api/src/crawler/engine.ts).
  6. Add a vitest spec under packages/api/src/__tests__/analyzer-my-check.test.ts.

Convention: issues are categorised by a single-word slug used in the dashboard's filter chips (page-titles, headings...).

Add a frontend route #

  1. Create packages/web/src/pages/MyNewPage.tsx. Use shared components (Card, Button, useApi).
  2. Register it inside the App.tsx router.
  3. If it's a per-run page, add a sidebar entry in sidebar.tsx.
  4. Add a Playwright spec that loads the route directly and asserts no console errors. The existing tests/e2e/spa-routes.spec.ts is the template.
Avoid mutating local storage directly in handlers. Use the helpers under lib/ (e.g. loadApiKeys(), saveAiSettings()).

AI provider #

  1. Add packages/api/src/ai/my-provider.ts with:
    export interface AiCall { provider: string; model: string; prompt: string; apiKey: string; }
    and an async call(req: AiCall): Promise<string>.
  2. Register the provider in packages/api/src/ai/index.ts.
  3. Add the provider entry to AI_PROVIDERS in packages/web/src/lib/ai-settings.ts.
  4. Settings → Insights now works end-to-end.

CLI command #

  1. Add packages/api/src/cli/commands/my-cmd.ts with a Commander definition.
  2. Register it from packages/api/src/cli/index.ts.
  3. Print a banner via printRoleBanner({ role: 'mycmd', stack: '...' }) so users see role-coded identity.
  4. For workflows, use printOk / printWarn / printError / printBullet from cli/banner.ts.

Tests #

The vitest config lives at packages/api/vitest.config.ts. The shared test fixture (packages/api/src/__tests__/setup.ts) opens a temporary SQLite in os.tmpdir(), creates one crawler run, and exposes makeRun, makeUrl, query.

import { describe, expect, it } from 'vitest';
import { analyzeTitles } from '../analyzer/titles.js';
import { makeRun, makeUrl, query } from './setup.js';

it('flags missing titles', () => {
  const run = makeRun('missing-title');
  makeUrl(run.id!, { address: 'https://example.com/no-title' });

  analyzeTitles(run.id!);

  const issues = query('SELECT type FROM issues WHERE crawl_run_id = ?', [run.id]);
  expect(issues.map(i => i.type)).toContain('missing_title');
});

The Playwright config sits at the workspace root (playwright.config.ts); specs live in tests/e2e/. The suite assumes a server on :7437; in CI you'd set PW_BOOT=1 to spawn pnpm start:sh.

Three categories today:

  • API smoke — direct request calls to /api/*.
  • Dashboard SPA — load the home page, click through navigation, count console errors.
  • SPA routes — parametrised loop that touches every top-level route.

When adding a new top-level route, append it to the loop in spa-routes.spec.ts.

Release #

  1. pnpm test:unit && PW_BOOT=1 pnpm test:e2e
  2. pnpm -r --filter './packages/*' build
  3. Commit in atomic, feature-grouped commits.
  4. git tag v0.X.Y and push.
The user-facing check is "does the dashboard render with zero console errors?" — that's exercised by e2e. Visual regressions are deliberately not asserted; the brand mark and sidebar are stable but tabs will evolve.

Code style #

  • Modular packages. New boundaries live in their own file under src/<feature>/; never cross-pollute siblings.
  • ESM everywhere. Both packages ship "type": "module". Use import/export with explicit .js suffixes.
  • Runtime vs dev split. Runtime deps live in the package that uses them; the workspace root owns only workspace-wide devDeps.
  • One grep-friendly constant. Paths the api runtime reads (public/, crawls/, exports/, tokens.json) all go through utils/workspace.ts.
  • Atomic commits. "Feature group per commit." If your change touches both api and web, split it before pushing.
Tests are part of the codebase. Don't skip them on a quick fix — a vitest call is < 200ms and an e2e run is < 5s.

Decisions #

  • Hono for the API. Small, fast, edge-friendly. Single port for API and SPA in production.
  • better-sqlite3 over libsql. Synchronous API fits a CLI-first tool; everything is a flat file under crawls/.
  • Playwright for JS rendering. First-party headless; auto-installed postinstall so users don't have to.
  • Vite + React for the SPA. HMR in dev; static build output out to public/ at the workspace root so the API server is the single source of truth for routes.
  • Monorepo keeps each side self-contained. Runtime deps live in @oseo/api or @oseo/web; the workspace root owns only devDeps and cross-cutting scripts.
  • Color-coded roles in the terminal. Backend process prints a solid blue ASCII banner; frontend process prints a solid magenta one. The orchestrator only prints a short startup message.
