1. Setup

$ pnpm install
$ pnpm --filter @oseo/api dev     # tsx watch backend at :7437 (blue BE banner)
$ pnpm --filter @oseo/web dev     # vite dev server at :5173 (magenta FE banner)
$ pnpm start:sh                  # both at once, short startup message + auto-install

# Quality
$ pnpm test:unit                  # vitest unit suite (api)
$ pnpm test:e2e                   # playwright spec (needs a server)
$ PW_BOOT=1 pnpm test:e2e         # spawn dev:sh, run, teardown

The orchestrator at the workspace root owns only the cross-cutting concerns (scripts, playwright config, tsconfig.base.json, pnpm- workspace.yaml). Runtime deps live in the package that needs them.

3. Adding a backend analyzer

  1. Add packages/api/src/analyzer/my-new-check.ts. Export a function named analyzeMyNewCheck(runId: number): void.
  2. Read URLs with getUrls(runId, { isInternal: true }) and produce an array of CrawlIssue objects.
  3. Call insertIssues(runId, issues) at the end.
  4. Export analyzeMyNewCheck from packages/api/src/analyzer/index.ts.
  5. Wire it from the engine after the URL has all the metadata you need (see packages/api/src/crawler/engine.ts).
  6. Add a vitest spec under packages/api/src/__tests__/analyzer-my-check.test.ts. The shared fixture opens a temp SQLite for you, so just call your analyzer and read issues.

Convention: issues are categorised by a single-word slug used in the dashboard's filter chips (page-titles, headings, ...).

4. Adding frontend route

  1. Create packages/web/src/pages/MyNewPage.tsx. Use the existing shared components — Card, Button, useApi — wherever you can.
  2. Add the route inside the App.tsx router.
  3. If the route is per-run, add a sidebar sub-entry in sidebar.tsx so power users can reach it via keyboard navigation.
  4. Add a Playwright spec that loads the route directly via page.goto() and asserts no console errors. The existing suite in tests/e2e/spa-routes.spec.ts is your template.
Avoid mutating local storage directly in handlers. Use the equivalent helper under lib/ (e.g. loadApiKeys(), saveAiSettings()) — that way the persistence shape and the migration list live in one place.

5. AI provider

  1. Add packages/api/src/ai/my-provider.ts exporting a function with the signature:
    export interface AiCall { provider: string; model: string; prompt: string; apiKey: string; }
    Implement async call(req: AiCall): Promise<string>.
  2. Register the provider in packages/api/src/ai/index.ts alongside the existing ones.
  3. Add the provider entry to AI_PROVIDERS in packages/web/src/lib/ai-settings.ts so the dropdown in Settings knows about it.
  4. Settings → Insights will work end-to-end with no further changes.

6. CLI command

  1. Add packages/api/src/cli/commands/my-cmd.ts with a Commander definition.
  2. Register it from packages/api/src/cli/index.ts.
  3. Print a banner via printRoleBanner({ role: 'mycmd', stack: '...' }) so users see the role-coded identity on every invocation.
  4. For workflow commands, use printOk / printWarn / printError / printBullet helpers from cli/banner.ts.

7. Vitest unit suite

The vitest config lives at packages/api/vitest.config.ts. The shared test fixture (packages/api/src/__tests__/setup.ts) opens a temporary SQLite database in os.tmpdir(), creates a single crawler run, and exposes helpers: 

import { describe, expect, it } from 'vitest';
import { analyzeTitles } from '../analyzer/titles.js';
import { makeRun, makeUrl, query } from './setup.js';

it('flags missing titles', () => {
  const run = makeRun('missing-title');
  makeUrl(run.id!, { address: 'https://example.com/no-title' });

  analyzeTitles(run.id!);

  const issues = query('SELECT type FROM issues WHERE crawl_run_id = ?', [run.id]);
  expect(issues.map(i => i.type)).toContain('missing_title');
});

Always write a unit spec per analyzer. The benchmarks don't run in CI by default, but the tool's correctness depends on the analyzer emitting the documented issues. If you change an analyzer, first run pnpm test:unit and update its tests.

8. Playwright e2e suite

The e2e config is at the workspace root (playwright.config.ts). Specs live under tests/e2e/. The suite assumes a server is bound on :7437; in CI you'd set PW_BOOT=1 to spawn pnpm start:sh automatically.

Three categories today:

  • API smoke — direct request calls to /api/*.
  • Dashboard SPA — pull up the home page, click through navigation, count console errors.
  • SPA routes — parametrised loop that touches every top-level route.

When adding a new top-level route, append it to the loop in spa-routes.spec.ts to keep coverage tight.

9. Release

  1. Run pnpm test:unit && PW_BOOT=1 pnpm test:e2e locally.
  2. Run pnpm -r --filter './packages/*' build.
  3. Commit everything in atomic, feature-grouped commits (the convention in this repo).
  4. Tag the commit with git tag v0.X.Y and push.
The user-facing check is "does the dashboard render with zero console errors?" — that gets exercised by the e2e suite. Visual regressions are deliberately not asserted; the brand mark + sidebar are stable but new tabs will evolve.

10. Code style

  • Modular packages. New boundaries live in their own file under src/<feature>/; never cross-pollute siblings.
  • ESM everywhere. Both packages ship "type": "module". Use import / export and add explicit .js suffixes to relative imports (TypeScript's ESM convention).
  • Runtime <=> dev split. Runtime deps live in the package that uses them; the workspace root owns only workspace-wide devDeps (typescript, tsx, vitest, eslint, concurrently, @playwright/test).
  • One grep-friendly constant. Paths the api runtime reads (public/, crawls/, exports/, tokens.json) all go through utils/workspace.ts. Don't reinvent that glob matching.
  • Atomic commits. The repo practices "feature group per commit". If your change touches both the api and the web, split it into two commits before pushing.
Tests are part of the codebase. Don't skip them on a quick fix — a vitest call is < 200 ms and an e2e run is < 5 s.
