1. Overview

Open SEO Checker is a single-process auditor that ships as a pnpm workspace with two real packages and a thin orchestrator at the root. One port is the production contract; the dev workflow adds a second.

  • @oseo/api — Hono + better-sqlite3, exposes the oseo CLI + REST API + SPA fallback. On startup it prints a solid-blue ASCII banner with the role label BACKEND.
  • @oseo/web — React 18 + Vite 6 SPA. Builds into /public at workspace root. Its Vite plugin prints a solid-magenta ASCII banner with the role label FRONTEND when the dev server starts.
  • Workspace orchestrator — start.sh at the workspace root + the root package.json scripts. It prints a short startup message and relies on the two child processes to print their own role-coloured banners.

2. Runtime topology

[ Backend ] Hono + SQLite ── serves (blue banner from server/app.ts) http://localhost:7437/ ─────────────────────────────────────► ├── /api/* → REST endpoints ├── /* → SPA fallback (serves public/index.html) └── /assets/* → static hashed bundles from public/ [ Frontend ] Vite dev server ── serves (magenta banner from vite.config.ts) http://localhost:5173/ ─────────────────────────────────────► ├── /api/* → proxied to backend on :7437 ├── /* → React SPA from packages/web/src/ └── HMR → live edits without reload

Production boots a single Node process with both responsibilities; the dev workflow splits them so HMR survives backend restarts. In dev mode the backend and frontend each own their startup banner, so the colour identity lives with the process that produces it.

3. Data flow

Browser Backend ──────── ──────── React SPA ────── GET /api/runs ───────► Hono route ▲ ◄───── JSON runs ─────── getAllRuns() │ (SELECT * FROM crawl_runs ...) │ React SPA ────── POST /api/crawl ─────► Hono route ▲ ◄───── JSON progress ─── CrawlEngine → SQLite crawl_runs.db │ WSS (fetchHtml(), analyzer, …) │ (optional) React SPA ────── GET /api/crawl/:id/issues ─► getIssues() ▲ ◄───── JSON issues ───── issues table → UI tab

Long-running crawls publish progress over the same HTTP channel (poll on /api/crawl/:id/status); a websocket bridge is optional and only used when the SPA requests it.

4. Crawl pipeline

[1] Seed URL → [2] FRONTIER queue │ ▼ [3] fetcher.fetchSingleHop(url) (recursive + cycle-detect) │ ▼ [4] Parser.parseHtml(body) ├── title, meta, headings, h1, h1-count ├── canonical, robots, hreflang ├── links (internal/external) ├── images (alt / dim / size) └── JSON-LD blocks │ ▼ [5] URL stored in SQLite (urls table) │ ▼ [6] Each analyzers/* runs for the run id: analyzeTitles → issues analyzeMeta → issues analyzeHeadings → issues analyzeLinks → issues analyzeImages → issues …

5. Redirect chain

fetcher.fetchSingleHop() recurses up to 10 hops with cycle detection on a per-host basis. The final response carries:

  • redirectChain — every URL we visited before getting a non-3xx response.
  • redirectUrl — the hop that ended the chain (often 301 → 200).
  • normalizedUrl — the address we ultimately parsed.

The engine stores one row per logical resource: the row's address is the resolved one, so duplicate detection (titles, meta descriptions) sees the canonical target. The chain itself is preserved as metadata so the UI can render a "redirects to" badge.

An earlier version of this code treated the 3xx body as the destination, which caused crawls to "see" exactly one row for many real-world sites. The recursive fetcher fixes it.

6. SQLite schema

The api opens a single database per crawl run and threads queries there.

TablePurpose
crawl_runsOne row per audit. Holds startUrl, status, started_at, JSON config and db_path.
urlsOne row per URL found. Holds status, title, meta description, headings, image counts, redirect chain, linked-by, linked-to.
linksEdges between URLs. source_url_id → target_url_id with anchor text + rel attributes.
imagesImage embeddings: src url, alt, width, height, size, status code.
issuesOne row per analyzer finding. Queryable by category, priority, status.

Indexes live on urls.crawl_run_id + status_category, issues.crawl_run_id + type, plus links.crawl_run_id + source_url_id — the heaviest queries (issue counts, URL list filters) hit the index path.

7. SPA boot

  1. Browser opens /; Hono falls back to public/index.html.
  2. React mounts into #root; the new AppLayout renders the shell (sidebar + topbar + main).
  3. Route resolution happens client-side via react-router-dom; no server round-trip for navigation.
  4. Each page issues its own useApi() requests — one per page, not one global query.
  5. API responses render until the next refresh; useInterval keeps a couple of pages (Dashboard, Overview) re-polling.

8. Workspace resolver

The api binary may be launched from many cwds: 

$ node packages/api/dist/index.js serve   # workspace root
$ oseo serve                                # pnpm --filter @oseo/api start
$ cd packages/api && node dist/index.js serve

To stay robust, packages/api/src/utils/workspace.ts walks upward from process.cwd() looking for pnpm-workspace.yaml. Every runtime path (public/, crawls/, exports/, tokens.json) flows through that helper, so the api resolves the same way no matter how it's launched.

10. Decision log

Why pnpm workspaces?

Single install + hoisted devDeps keep the api lean and avoid cross-package node_modules drift. pnpm-workspace.yaml declares both packages; the root owns scripts only.

Why Hono, not Express?

Composability (typed routes, middleware as functions) and edge-deploy compatible. SQLite-via-better-sqlite3 gives us Node-side analytical queries in < 200 ms.

Why recursive fetcher, not undici maxRedirections?

Unreachable for the chromium step; we needed cycle detection and the full chain to show up in the UI as one row per logical destination.

Why natural page scroll, not viewport-locked layout?

Playwright's fullPage captures the whole document; locking <main> into its own scroll container produced "blank bottom" screenshots for everyone debugging report images.

Why AIProvider catalogue in the SPA, not just the API?

Provider key + active selection live in browser local storage because the dashboard is single-tenant local-first. The api only sees whatever the request carries.

Why atomic feature-grouped commits?

Each change is a tiny, reviewable story. git reset --mixed HEAD~1 + restage is faster than rewriting history when a commit accidentally mixes concerns.
